# 技能撰写最佳实践

> 学习如何撰写能被 Claude 发现并成功使用的技能。

优秀的技能简洁、结构清晰，并经真实使用验证。本指南给出实用撰写决策，帮助你写出可被有效发现的技能。概念背景见官方 [Skills 概览](/en/docs/agents-and-tools/agent-skills/overview)。

**说明：** 英文原版已另存为同目录下的 `anthropic-best-practices.source-en.md`，便于与上游对照。

## 核心原则

### 简洁是关键

[上下文窗口](https://platform.claude.com/docs/en/build-with-claude/context-windows)是公共资源。技能与系统提示、对话历史、其他技能的元数据、用户请求共享同一窗口。

启动时通常只预加载各技能的元数据（`name` 与 `description`）。相关时才会读 `SKILL.md`，并按需读其他文件。但一旦加载 `SKILL.md`，每个 token 都与历史等竞争——仍要简洁。

**默认假设：** Claude 已经很聪明。

只补充它尚不具备的上下文。对每条信息追问：「真的需要解释吗？」「能否假定它已知？」「这段是否值回 token？」

**自由度：** 按任务脆弱性与变化性匹配具体程度——高自由度给文字指引；中自由度给带参数的伪代码/脚本；低自由度给少参或无参的固定脚本（易错、需一致、必须按序的操作）。类比：窄桥悬崖边要给精确护栏（低自由度）；开阔平地可给方向让模型选路（高自由度）。

### 在计划使用的所有模型上测试

技能是对模型的补充，效果依赖底层模型。在计划使用的每个模型上都测试。

- **Haiku**：指引是否足够？  
- **Sonnet**：是否清晰高效？  
- **Opus**：是否避免过度解释？  

跨模型使用时，取能兼顾各模型的写法。

## 技能结构

**YAML Frontmatter：** `SKILL.md` 需要 `name`（最多约 64 字符人读名）与 `description`（单行说明做什么、何时用，最多 1024 字符）。完整结构见 [Skills 概览](https://agentskills.io/specification) 与官方文档。

### 命名惯例

推荐用**动名词**（verb + -ing）表达活动或能力，如「Processing PDFs」「Testing code」。也可用名词短语或动宾短语。避免「Helper」「Utils」等空泛名。集合内命名风格保持一致。

### 撰写有效的 description

**始终第三人称**（description 会注入系统提示）。

- 好：「处理 Excel 并生成报表」  
- 差：「我可以帮你处理 Excel」  

要**具体并含关键词**。`description` 用于从大量技能中做选择；正文才是实现细节。每条技能只有一段 description，必须足以让 Claude 判断「该不该选我」。

（**注意：** Superpowers 的 `writing-skills/SKILL.md` 对 description 有额外约束：只写触发条件、不写流程概括，以免模型只读 description 跳过正文。若同时遵循本文与 Superpowers，以 Superpowers 仓库准则为准。）

### 渐进披露模式

- **由简到繁：** 先概述再链到高级文档。  
- **模式 1：** 高层指南 + 多个参考文件（快速开始 / 高级 / API）。  
- **模式 2：** 按领域分子目录（如 BigQuery 各数据集一篇）。  
- **模式 3：** 条件细节——「若需 X 见 advanced.md」。  

避免**嵌套过深的引用**：引用只从 `SKILL.md` **一层**链出去；否则 Claude 可能只 `head` 预览，信息不全。

**长参考文件（100+ 行）** 顶部加目录（Contents），便于预览时看到全貌。

## 工作流与反馈环

### 复杂任务用工作流

拆成清晰顺序步骤；很复杂时可给清单，让 Claude 复制并在回复里勾选。

示例包括：无代码的研究综合流程；有代码的 PDF 填表流程（每步对应脚本）。清单帮助双方跟踪进度。

### 实现反馈环

常见模式：**运行校验器 → 修错 → 重复**。可针对样式指南（无代码）或 XML 校验脚本（有代码）。尽早验证可大幅提高输出质量。

## 内容准则

### 避免易过时信息

不要写「2025 年 8 月前用旧 API」这类会过期的话。改用「当前方法」+「旧模式」折叠区保留历史。

### 术语一致

全文同一概念用同一词（如始终「API endpoint」），不要混用 URL/route 等混称。

## 常见模式

### 模板模式

为输出格式提供模板。要求严（API 形态）时用固定结构；允许变通时给默认格式并说明可按情境调整。

### 示例模式

依赖「看例子才懂风格」的技能，提供多组输入/输出对（如提交信息格式）。

### 条件工作流

在决策点分支：新建走 A 流程，编辑走 B 流程。工作流过长可拆到单独文件并按任务指示阅读。

## 评估与迭代

### 先建评估

**先写大量文档前，先建评估。** 确保技能解决真问题，而不是想象需求。

评估驱动开发步骤：找缺口 → 建 3 个左右场景 → 无技能测基线 → 写最少说明通过评估 → 迭代。

评估可用 JSON 描述 query、files、expected_behavior 等（示例见英文原文）。当前无内置运行器，需自建评估管线。

### 与 Claude 迭代开发技能

用「Claude A」写技能，「Claude B」在真实任务中试用。流程：不用技能完成一次任务，记录重复提供的上下文 → 抽象可复用模式 → 让 A 生成技能 → 删冗余、改信息架构 → B 在类似任务上测 → 带回具体失败再改。

改进现有技能：B 真实使用 → 观察卡点 → 回 A 改措辞/结构 → 再测。

收集团队反馈：是否易被触发、说明是否清晰、缺什么。

### 观察 Claude 如何浏览技能

注意：阅读顺序是否符合预期、是否漏掉重要链接、是否反复读同一文件（是否应并入正文）、是否从不打开某文件（是否多余或信号不足）。`name`/`description` 对触发尤其关键。

## 应避免的反模式

### 不要用 Windows 风格路径

始终用正斜杠：`scripts/helper.py`，即使在 Windows 上。反斜杠在 Unix 会出错。

### 不要堆砌过多选项

不要列一长串可选库。给**默认**（并说明例外何时用别的），例如默认 `pdfplumber`，扫描件再 OCR 用另一套。

## 进阶：含可执行代码的技能

若技能只有 Markdown，可跳到文末检查清单。

### 解决，不要甩锅

脚本里处理错误，不要 `open(path)` 一抛了之。文件不存在可创建默认、权限问题可降级。配置常量要注释说明理由，避免「魔法数字」。

### 提供实用脚本

预置脚本比每次让 Claude 写更稳、更省 token、更一致。说明是**执行**（`Run analyze_form.py`）还是**当参考读**（复杂算法）。多数情况执行更合适。详见下文运行环境。

### 视觉分析

输入可渲染成图时，可先转图片再让 Claude 看布局（需自备如 `pdf_to_images.py`）。

### 可验证的中间产物

复杂批处理用「计划 → 校验 → 执行」：先写 `changes.json` 等，经脚本验证再应用，早错早改。适合批量、破坏性强、规则复杂的操作。校验信息要具体（列出可用字段名）。

### 包依赖

- **claude.ai** 执行环境：可从 npm/PyPI/GitHub 安装。  
- **Anthropic API**：无网络、不能运行时装包。  

在 `SKILL.md` 列出依赖并对照[代码执行工具文档](/en/docs/agents-and-tools/tool-use/code-execution-tool)确认可用。

### 运行环境要点

技能在带文件系统、bash、代码执行的环境中运行。要点：

1. 启动时预加载各技能 YAML 元数据。  
2. 按需用工具读 `SKILL.md` 等。  
3. 脚本可通过 bash 执行，**不必**把整份脚本塞进上下文，只有输出耗 token。  
4. 大参考文件在读之前不占上下文。  

路径用正斜杠；文件名表意（`form_validation_rules.md`）；按领域组织目录；确定性操作用脚本；执行意图写清。示例目录结构见英文原文 `bigquery-skill/` 示例。

更完整架构说明见 [Skills 概览 — How Skills work](/en/docs/agents-and-tools/agent-skills/overview#how-skills-work)。

### MCP 工具引用

使用**完全限定名**：`ServerName:tool_name`，避免多 MCP 时找不到工具。

### 不要假定已安装包

写明 `pip install ...` 再 `import`，不要只说「用 pdf 库」。

## 技术说明

### YAML frontmatter

`name` 与 `description` 长度限制见官方文档；完整字段以 [规范](https://agentskills.io/specification) 为准。

### Token 预算

`SKILL.md` 正文宜控制在约 500 行以内；更长则用渐进披露拆文件。

## 有效技能检查清单

### 核心质量

- [ ] description 具体且含关键词  
- [ ] description 说明**做什么**与**何时用**（若与 Superpowers CSO 冲突，以项目准则为准）  
- [ ] 正文约 500 行内或已拆分  
- [ ] 无易过时信息（或已放入「旧模式」）  
- [ ] 术语一致  
- [ ] 示例具体  
- [ ] 引用一层深  
- [ ] 渐进披露得当  
- [ ] 工作流步骤清楚  

### 代码与脚本

- [ ] 脚本解决问题而非甩给 Claude  
- [ ] 错误处理明确  
- [ ] 无常数无解释  
- [ ] 依赖已列出且环境支持  
- [ ] 脚本有说明  
- [ ] 路径全用正斜杠  
- [ ] 关键操作有校验/验证环  

### 测试

- [ ] 至少三个评估场景  
- [ ] 在 Haiku / Sonnet / Opus 上试过  
- [ ] 真实场景测过  
- [ ] 团队反馈已吸收（若有）

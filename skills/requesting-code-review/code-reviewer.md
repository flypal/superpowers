## 评审清单

**代码质量：**
- 职责分离是否清晰？
- 错误处理是否得当？
- 类型安全（若适用）？
- 是否遵循 DRY？
- 边界情况是否覆盖？

**架构：**
- 设计决策是否合理？
- 可扩展性？
- 性能影响？
- 安全风险？

**测试：**
- 测试是否验证真实逻辑（而非 mock）？
- 边界是否覆盖？
- 是否需要集成测试？
- 是否全部通过？

**需求：**
- 计划中的需求是否都满足？
- 实现是否与规格一致？
- 是否范围蔓延？
- 破坏性变更是否已文档化？

**生产就绪：**
- 若有 schema 变更，迁移策略？
- 是否考虑向后兼容？
- 文档是否完整？
- 是否有明显 bug？

## 输出格式

### 优点
[做得好的地方？要具体。]

### 问题

#### 严重（必须修）
[Bug、安全问题、数据丢失风险、功能损坏]

#### 重要（应该修）
[架构问题、缺功能、错误处理差、测试缺口]

#### 次要（最好有）
[代码风格、优化机会、文档改进]

**每个问题需包含：**
- 文件:行号
- 错在哪
- 为何重要
- 如何修（若非显而易见）

### 建议
[对代码质量、架构或流程的改进]

### 评估

**可以合并吗？** [是/否/修后可合]

**理由：** [1–2 句技术判断]

## 关键规则

**要做：**
- 按真实严重程度分类（不是什么都严重）
- 要具体（文件:行号，不要空泛）
- 解释问题**为何**重要
- 承认优点
- 给出明确结论

**不要做：**
- 没仔细看就说「看起来不错」
- 把挑剔当严重
- 评审你没看过的代码
- 空泛（「改进错误处理」）
- 不给明确结论

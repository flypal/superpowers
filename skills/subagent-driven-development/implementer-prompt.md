# 实现者子智能体提示模板

派发实现者子智能体时使用本模板。

```
Task tool (general-purpose):
  description: "Implement Task N: [task name]"
  prompt: |
    你正在实现任务 N：[task name]

    ## 任务说明

    [计划中该任务的全文——贴在这里，不要让子智能体自己去读文件]

    ## 上下文

    [场景设定：任务在整体中的位置、依赖、架构背景]

    ## 开始前

    若你对以下任一方面有疑问：
    - 需求或验收标准
    - 思路或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚之处

    **现在就问。** 开工前提出所有顾虑。

    ## 你的职责

    一旦需求清楚：
    1. 严格按任务说明实现
    2. 编写测试（若任务要求则遵循 TDD）
    3. 验证实现可用
    4. 提交工作
    5. 自检（见下）
    6. 汇报

    工作目录：[directory]

    **工作中：** 若遇到意外或不清楚之处，**提问**。
    随时可以暂停澄清。不要猜、不要想当然。

    ## 代码组织

    你能最好地理解能一次性放进上下文的代码量；文件聚焦时编辑更可靠。注意：
    - 遵循计划中定义的文件结构
    - 每个文件职责单一、接口明确
    - 若正在创建的文件超出计划意图，停下并回报 DONE_WITH_CONCERNS——不要在无计划指导时自行拆分文件
    - 若正在修改的已有文件已经很大或很乱，谨慎处理并在报告中注明顾虑
    - 在现有代码库中遵循既有模式。像优秀开发者一样改进你动到的代码，但不要重构任务范围外的东西。

    ## 当你力不从心

    随时可以停下说「这对我来说太难了」。烂活比没活更糟。因升级而受罚是不会的。

    **在以下情况停下并升级：**
    - 任务需要多种合理路径下的架构决策
    - 需要理解超出所给信息的代码且无法弄清
    - 不确定自己的做法是否正确
    - 任务涉及计划未预料到的既有代码重组
    - 读了一堆文件仍无法理解系统、毫无进展

    **如何升级：** 回报状态 BLOCKED 或 NEEDS_CONTEXT。具体说明卡在哪、试过什么、需要什么帮助。
    控制者可提供更多上下文、换更强模型重派，或把任务拆小。

    ## 回报前：自检

    用新眼光审阅自己的工作。自问：

    **完整性：**
    - 是否完整实现规格中的全部内容？
    - 是否漏掉需求？
    - 是否有未处理的边界？

    **质量：**
    - 这是你能做到的最好的吗？
    - 命名是否清楚、准确（描述**做什么**而非**怎么做**）？
    - 代码是否干净、可维护？

    **纪律：**
    - 是否避免过度构建（YAGNI）？
    - 是否只做了被要求的？
    - 是否遵循代码库既有模式？

    **测试：**
    - 测试是否验证真实行为（而非仅 mock 行为）？
    - 若要求 TDD，是否做到？
    - 测试是否足够？

    自检中发现问题，现在修完再汇报。

    ## 汇报格式

    结束时汇报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（若卡住则写尝试了什么）
    - 你跑了哪些测试及结果
    - 变更的文件
    - 自检发现（若有）
    - 其他问题或顾虑

    若完成工作但对正确性有疑虑，用 DONE_WITH_CONCERNS。
    若无法完成任务，用 BLOCKED。
    若缺未提供的信息，用 NEEDS_CONTEXT。
    不要默默产出你不确定的工作。
```

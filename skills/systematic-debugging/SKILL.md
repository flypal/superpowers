# 系统化调试

## 概述

随机打补丁浪费时间，还会制造新 bug。快速掩盖往往掩盖根本问题。

**核心原则：** **始终**先找根因再谈修复。只治症状就是失败。

**违反流程的字面，就是违反调试的精神。**

## 铁律

```
未完成第 1 阶段根因调查，不得提出修复
```

若第 1 阶段未完成，就不能提议改代码。

## 何时使用

适用于**任何**技术问题：
- 测试失败  
- 生产 bug  
- 意外行为  
- 性能问题  
- 构建失败  
- 集成问题  

**在以下情况尤其要用：**
- 时间压力下（紧急时最容易猜）  
- 「就一行快修」看起来很明显  
- 已经试过多种修复  
- 上次修复无效  
- 尚未完全理解问题  

**不要在以下情况跳过：**
- 问题看起来简单（简单 bug 也有根因）  
- 你很赶（赶工保证返工）  
- 经理要「马上修好」（系统化比乱试更快）  

## 四个阶段

你必须完成每一阶段才能进入下一阶段。

### 第 1 阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**  
   - 不要跳过错误或警告  
   - 答案经常在错误里  
   - 读完整个栈  
   - 记下行号、路径、错误码  

2. **稳定复现**  
   - 能否可靠触发？  
   - 精确步骤是什么？  
   - 是否每次发生？  
   - 若无法复现 → 多收集数据，不要猜  

3. **检查近期变更**  
   - 什么改动可能导致？  
   - git diff、近期提交  
   - 新依赖、配置变化  
   - 环境差异  

4. **多组件系统中收集证据**  

   **当系统有多层（CI → 构建 → 签名，API → 服务 → 数据库）时：**  

   **在提议修复之前，先加诊断埋点：**  
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 核对环境/配置是否传递
     - 检查每一层的状态

   运行一次收集证据，看清**断在哪一层**
   再分析证据，定位失败组件
   再针对该组件深入
   ```

5. **追踪数据流**  

   **当错误深埋在调用栈中：**  

   本目录的 `root-cause-tracing.md` 有完整回溯技巧。  

   **简版：**  
   - 错误值从哪来？  
   - 谁用错误值调了这里？  
   - 一直往上追到源头  
   - 在**源头**修，不在症状处修  

### 第 2 阶段：模式分析

**先找模式再修：**

1. **找正常工作的例子**  
   - 在同代码库找类似且正常的代码  
   - 什么能工作、什么坏了？  

2. **对照参考实现**  
   - 若在实现某种模式，**完整**阅读参考实现  
   - 不要略读——逐行读  
   - 在套用前完全理解模式  

3. **找出差异**  
   - 正常与异常之间有什么不同？  
   - 列出所有差异，再小也别假设「那不可能重要」  

4. **理解依赖**  
   - 还需要哪些组件？  
   - 哪些设置、配置、环境？  
   - 有哪些隐含假设？  

### 第 3 阶段：假设与验证

**科学方法：**

1. **形成单一假设**  
   - 清楚写出：「我认为根因是 X，因为 Y」  
   - 写下来  
   - 要具体，不要空泛  

2. **最小化验证**  
   - 用**最小**改动验证假设  
   - 一次只变一个变量  
   - 不要同时修多处  

3. **继续前先验证**  
   - 有效？→ 第 4 阶段  
   - 无效？→ 形成**新**假设  
   - **不要**在失败假设上再叠修复  

4. **不知道时**  
   - 明说「我不懂 X」  
   - 不要装懂  
   - 求助  
   - 再研究  

### 第 4 阶段：实现

**修根因，不治症状：**

1. **先写失败测试用例**  
   - 最简单复现  
   - 尽量自动化  
   - 无框架则写一次性脚本  
   - 修复**前**必须有  
   - 写正式失败测试用 `superpowers:test-driven-development` 技能  

2. **单一修复**  
   - 针对已识别的根因  
   - **一次**只改一处  
   - 不要「顺手」重构  
   - 不要捆绑重构  

3. **验证修复**  
   - 测试现在通过？  
   - 其他测试仍通过？  
   - 问题真解决了？  

4. **若修复无效**  
   - **停**  
   - 数一下：已经试过几次修复？  
   - 若 < 3：回到第 1 阶段，用新信息重分析  
   - **若 ≥ 3：停，质疑架构（见下第 5 步）**  
   - **不要**在未经架构讨论的情况下尝试第 4 次修复  

5. **若 3 次以上失败：质疑架构**

   **暗示架构问题的模式：**  
   - 每次修复都在新位置暴露共享状态/耦合等问题  
   - 修复需要「大规模重构」才能实现  
   - 每次修复都在别处制造新症状  

   **停，质疑基本假设：**  
   - 这个模式根本上成立吗？  
   - 我们是否靠惯性硬撑？  
   - 应重构架构还是继续治症状？  

   **与伙伴讨论后再尝试更多修复**  

   这不是「假设失败」——这是**架构错了**。

## 危险信号 — 停，按流程来

若你发现自己这样想：
- 「先快修，以后再查」  
- 「改改 X 看行不行」  
- 「多处一起改，跑测试」  
- 「跳过测试，我手动验」  
- 「大概是 X，我修一下」  
- 「不太懂，但可能有用」  
- 「模式说 X，但我换个用法」  
- 「主要问题有这些：[列修复] 但没调查」  
- 在追踪数据流之前就提方案  
- **「再试一次修复」（已经失败 2+ 次）**  
- **每次修复都在不同地方冒新问题**  

**以上全部意味着：停。回到第 1 阶段。**

**若 3+ 次修复失败：** 质疑架构（见第 4 阶段第 5 步）

## 伙伴发出的「你做错了」信号

**注意这些纠偏：**
- 「是不是没发生？」—— 你假设了却没验证  
- 「能不能让我们看到……？」—— 你本该加证据收集  
- 「别猜」—— 你在没理解的情况下提修复  
- 「好好想」—— 质疑基本假设，不止症状  
- 「我们卡住了？」（沮丧）—— 你的方法不对  

**看到这些：** 停。回到第 1 阶段。

## 常见合理化

| 借口 | 事实 |
|--------|---------|
| 「问题简单，不需要流程」 | 简单问题也有根因。简单 bug 走流程也很快。 |
| 「紧急，没时间走流程」 | 系统化比乱试更快。 |
| 「先试一下，再调查」 | 第一次修复定调。一开始就做对。 |
| 「确认有效后再写测试」 | 未经验证的修复站不住。先有测试才证明。 |
| 「多处一起改省时间」 | 无法隔离哪条有效。还会引入新 bug。 |
| 「参考太长，我变通一下」 | 一知半解必出 bug。要读完。 |
| 「我看到问题了，我来修」 | 看到症状 ≠ 理解根因。 |
| 「再试一次」（已失败 2+ 次） | 3+ 次失败 = 架构问题。质疑模式，别再瞎修。 |

## 速查

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错、复现、查变更、收集证据 | 理解**是什么、为什么** |
| **2. 模式** | 找正常例子、对比 | 找出差异 |
| **3. 假设** | 立论、最小验证 | 确认或新假设 |
| **4. 实现** | 写测试、修复、验证 | bug 解决，测试通过 |

## 当流程显示「没有根因」

若系统调查后问题确实是环境、时序或外部因素：

1. 你已走完流程  
2. 记录调查过程  
3. 实现合适处理（重试、超时、错误信息）  
4. 加监控/日志供日后查  

**但是：** 声称「没有根因」的案例里约 95% 是调查不完整。

## 辅助技巧

下列技巧属于系统化调试，见本目录：

- **`root-cause-tracing.md`** — 沿调用栈回溯找最初触发点  
- **`defense-in-depth.md`** — 找到根因后在多层加校验  
- **`condition-based-waiting.md`** — 用条件轮询替代任意超时  

**相关技能：**
- **superpowers:test-driven-development** — 写失败测试用例（第 4 阶段第 1 步）  
- **superpowers:verification-before-completion** — 声称成功前先验证修复  

## 实际影响

来自调试会话：
- 系统化：约 15–30 分钟修好  
- 乱试：常折腾 2–3 小时  
- 一次修对率：约 95% vs 40%  
- 引入新 bug：几乎为零 vs 常见
